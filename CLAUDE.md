# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo shape

pnpm + Turborepo monorepo. Two apps share a Postgres backend:

- `apps/web` — Next.js 15 App Router (React 19, RSC). The whole product backend lives here: Auth.js, Gmail OAuth, cron workers, billing, JSON APIs for iOS.
- `apps/ios` — Native SwiftUI client, iOS 17+. `Kyujin.xcodeproj` is generated by **xcodegen** from `project.yml` — never hand-edit it.
- `packages/db` — Drizzle schema + Postgres client (`@kyujin/db`). Single source of truth for the data model.
- `packages/shared` — Gmail SDK wrapper, email classifier, prompts, sender-domain lists, ATS extractor, types (`@kyujin/shared`). Imported by `apps/web` server code; not used by iOS.

**Brand note:** the web app is mid-rebrand from "Kyujin" → "Yume" (production domain `yumeai.app`). The repo, npm package names (`@kyujin/*`), and iOS app/bundle stay Kyujin. iOS App Store display name is "Kyujin: Job Tracker". When editing UI copy, prefer Yume; when editing package names, schema, or iOS code, prefer Kyujin.

## Commands

From the repo root:

```bash
pnpm install
pnpm dev          # starts apps/web on http://localhost:3100 (NOT 3000 — README is wrong)
pnpm build        # turbo build, all workspaces
pnpm lint
pnpm typecheck    # turbo run typecheck — runs `tsc --noEmit` per workspace

pnpm db:push      # apply Drizzle schema to DATABASE_URL (dev)
pnpm db:migrate   # run committed migrations (prod / CI)
pnpm db:studio    # drizzle-kit studio

pnpm ios:open     # cd apps/ios && xcodegen && open Kyujin.xcodeproj
pnpm ios:gen      # regenerate xcodeproj without opening
```

Web app scripts use Next/Turbopack and port **3100** (`apps/web/package.json`). `AUTH_URL`, `GMAIL_REDIRECT_URI`, and `NEXT_PUBLIC_APP_URL` in `.env.example` all reflect 3100 — keep them in sync if you change the port.

There is **no test suite** in this repo. "Verification" is manual; see README "Verification (manual)" for the dev flow.

## Architecture

### Email ingestion pipeline (the core of the product)

Clients never trigger per-message processing. Ingestion is always server-side. The only mailbox path is Gmail-direct, talking to Google with our own OAuth client and `gmail.readonly` scope — going public requires Google CASA Tier-2 audit on our own Google Cloud project.

1. **Backfill or webhook** drops message IDs into the `backfill_queue` table.
   - `POST /api/gmail/backfill` enqueues N days of history (90 default; 365 is premium-gated).
   - `POST /api/gmail/pubsub` receives Gmail push notifications, diffs `historyId`, enqueues new IDs. Uses a shared-secret query-param token, not a session — must be excluded from auth middleware (see "Middleware split" below).
2. **Vercel Cron** hits `GET /api/cron/process-batch` every 5 minutes (`vercel.ts` at repo root defines the schedule).
3. The worker pulls up to **50** queued messages, fetches each via the Gmail SDK, normalizes (`normalizeGmailMessage`), runs `preClassify` (sender filter / handshake regex / etc.), then `classifyLlm` for survivors, then `upsertApplicationFromClassification`.
4. **Classifier:** `packages/shared/src/classifier.ts` uses `@ai-sdk/google` directly with `gemini-2.5-flash-lite`, reading `GOOGLE_GENERATIVE_AI_API_KEY`. **Not the Vercel AI Gateway** — `AI_GATEWAY_API_KEY` is in the env list but the active path doesn't use it.
5. A second cron `GET /api/cron/refresh-watches` runs every 12 hours to re-issue Gmail `users.watch` subscriptions before they expire (Gmail watches die after 7 days).

Both cron routes auth via `isAuthorizedCron` (Bearer `CRON_SECRET`). `maxDuration = 300` is set on `process-batch` to use the long-running Vercel function limit.

### Auth: two parallel paths

- **Web (cookies):** Auth.js v5 with JWT session strategy + Drizzle adapter. Login is Google (required) + Apple (optional, auto-disabled if env vars blank). User row is created on first OAuth callback — no separate sign-up form.
- **iOS (bearer tokens):** Sign in with Apple natively → backend mints an opaque session token written to the same `sessions` table that Auth.js uses → iOS sends `Authorization: Bearer <token>` on every request.

**`apps/web/lib/api-auth.ts → getAuthUserId(req)` is the unified resolver.** Any API route that should accept both clients must call it (cookie first, then Bearer fallback). Routes that only serve the web (e.g. `/api/gmail/*`) can rely on middleware.

### Middleware split (Edge vs Node)

There are **two Auth.js configs**, and this is load-bearing:

- `apps/web/auth.config.ts` — slim, Edge-safe, no DB adapter, JWT-only. Imported by `middleware.ts`.
- `apps/web/auth.ts` — full config, spreads `auth.config.ts`, adds the DrizzleAdapter. Used by route handlers / server components on the Node runtime.

`middleware.ts` uses an **explicit allowlist matcher** (`/app/:path*` plus the cookie-protected Gmail routes: `/api/gmail/{backfill,connect,disconnect,watch}`). Routes that auth themselves must stay off this list:

- `/api/gmail/pubsub` — auths via query-param token. Adding it to the matcher caused an infinite redirect loop with Google Pub/Sub (see commit `5c077d0`).
- `/api/gmail/callback` — auths via signed OAuth state nonce.
- `/api/applications`, `/api/stats` — accept Bearer tokens, can't go through cookie-only redirect.

If you add a new API route, decide its auth strategy first, then choose whether to add it to the middleware matcher or call `getAuthUserId` inside the handler.

### Gmail: two OAuth clients

`AUTH_GOOGLE_*` (login, `openid email profile`) and `GMAIL_*` (data access, `gmail.readonly`) are **separate OAuth clients** by design. Login doesn't request Gmail scopes; the Gmail grant is opt-in from `/app/settings` so Apple users can also connect Gmail. `gmail.readonly` is a restricted scope — going public requires Google CASA Tier-2 audit on our own Google Cloud project.

### Database (`packages/db/src/schema.ts`)

Authoritative model. Notable invariants when writing queries / migrations:

- **Application identity** has three tiers: `(userId, company, role)` unique, `(userId, matchKey)` unique (normalized form, computed in JS at insert), and `(userId, jobId)` indexed. `matchKey` exists specifically so concurrent crons can't split the same job into two rows. Touch these with care.
- **Billing is dual-source**: Stripe (`stripe*` columns) for web, Apple StoreKit (`apple*` columns) for iOS. `trialUsedAt` is shared bookkeeping so a user can't trial on one platform and again on the other. The web's `plan` column is the read source-of-truth; the `*_status` columns reflect provider state.
- **Status enum** `application_status`: `applied | no_response | interview | rejected | accepted | obtained`. Used both for applications and for individual classifications.
- **Audit log** `application_audit` records reversible user actions (merge/move/detach) with enough payload to undo. `revertedAt` prevents double-undo.
- **Usage caps**: `chat_usage` (daily, AI chat) and `classifier_usage` (rolling-30d, LLM classifier calls). Both are append-only.

### Shared package boundary

`@kyujin/shared` re-exports everything from `packages/shared/src/index.ts`. Subpath exports (`./classifier`, `./gmail`, etc.) are declared in its `package.json` and consumed by the web app. The package is **server-side only** — it imports `googleapis`, `@ai-sdk/google`, `drizzle-orm`. Never import `@kyujin/shared` from a client component.

`apps/web/next.config.ts` has `transpilePackages: ['@kyujin/db', '@kyujin/shared']` and `serverExternalPackages: ['postgres', 'googleapis']` — required for the workspace setup to build.

### iOS ↔ web coupling

The iOS app expects backend endpoints (`/api/auth/ios-apple` for token exchange, `/api/applications`, `/api/stats`) that are partially in flight — check `apps/ios/README.md` "TODOs" section before assuming an endpoint exists. Wire types live in `apps/ios/Kyujin/Models/Application.swift` and must be kept in lockstep with the JSON returned by `apps/web` handlers — there is no code-gen.

When working in `apps/ios`, prefer XcodeBuildMCP tools over `xcodebuild` shell invocations. The xcodeproj is generated, so changes to project structure go in `apps/ios/project.yml` then `pnpm ios:gen`.

## Deployment

Vercel-only. `vercel.ts` at the repo root defines build/install/ignore commands and the cron schedule — Vercel reads it on every deploy. The `ignoreCommand` skips builds when only files outside `apps/web` and `packages` changed.

Database is **Supabase Postgres** (pooled connection via `aws-1-us-west-1.pooler.supabase.com:6543`). `DATABASE_URL` is set as a Vercel env (not auto-injected by a Marketplace integration) and pulled locally via `vercel env pull`. Other secrets live in `vercel env`.

Note on `drizzle-kit migrate`: the local `__drizzle_migrations` ledger has drifted from the journal file (mid-numbered tags applied out of order at some point), so `pnpm db:migrate` reports success without applying newer entries. Until that's reconciled, apply new migrations by piping the raw SQL through `psql "$DATABASE_URL"` — the schema lands correctly, just without a ledger row. The drizzle-orm bump follow-up is a good time to also reconcile this.
